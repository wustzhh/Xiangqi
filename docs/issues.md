# 待修复问题清单

## 1. 手机连接不上服务器

**描述**：手机端无法连接到远程 VPS `212.129.243.158:8080`。电脑端可以正常连接，说明服务器运行正常，但手机网络（移动数据/WiFi）可能屏蔽了 8080 端口。

**可能原因**：
- 运营商封锁非标准端口
- VPS 防火墙未开放 8080
- 需要 WSS（WebSocket Secure）而非 WS

**解决方案**：
- 方案 A：在 VPS 上用 Nginx 反向代理，将 `wss://212.129.243.158/ws` 转发到 `ws://127.0.0.1:8080`，客户端改用 443 端口 + wss 协议
- 方案 B：服务器改用 443 或 80 端口（需管理员权限）
- 方案 C：客户端设置页面允许用户自定义服务器地址（已有该功能）

## 2. 无将死/困毙判定

**描述**：游戏中没有将军、将死和困毙的自动判定。目前只有"认输"和"对方离开"两种方式结束游戏。先手玩家可以将死对方但游戏不会自动结束。

**影响范围**：
- 服务器：`server/src/room_manager.dart` 的 `handleMove` 只广播走法，不做棋局判定
- 客户端：`lib/screens/room_screen.dart` 没有将死检测逻辑

**解决方案**：
- 在客户端 `room_screen.dart` 每次 `move_made` 后调用 `rules.isCheckmate()` 和 `rules.isStalemate()`
- 如果检测到将死/困毙，发送 `game_over` 消息到服务器
- 或者把棋局逻辑搬到服务器端做权威判定

## 3. 认输流程需验证

**描述**：认输功能已实现（`handleResign`），但未充分测试。认输后 `game_over` 广播，赢家能看到胜利画面，但返回大厅后的房间列表状态可能不对。

**需要验证**：
- 认输后双方都能正确返回大厅
- 房间在游戏结束后正确关闭或重置
- 返回大厅后房间列表不显示"进行中"的房间

## 4. 断线重连后棋盘状态未恢复

**描述**：断线重连功能已实现（`tryReconnect`），但重连后只发了 `roomJoined` 和 `gameStarted` 标记，没有发送当前的棋盘状态（走棋历史）。重连的玩家会看到初始棋盘而不是实际对局位置。

**解决方案**：
- 服务器需要保存走棋历史列表
- 重连时将完整的历史走法发给重连的玩家
- 客户端重放所有走法重建棋盘

## 5. Windows 客户端文件被占用

**描述**：`applications/windows-client/` 目录下的文件在程序运行时无法覆盖。每次更新需要先关闭旧程序再复制新文件。

**解决方案**：复制的命令行操作前先执行 `taskkill /f /im xiangqi.exe`。

## 6. applications/xiangqi.apk 体积过大

**描述**：APK 文件 52MB，接近 GitHub 的 50MB 文件大小限制。推送时会出现警告。

**解决方案**：
- 将 `applications/*.apk` 和 `applications/windows-client/` 加入 `.gitignore`
- 构建产物在 `build/` 目录中已有，`applications/` 只是复制品，可忽略

## 7. 房间设置未保存到服务器

**描述**：当前房间设置（悔棋/步时/局时/先后手）只在内存中，服务器重启后丢失。另外"步时"和"局时"倒计时功能尚未真正实现——界面上可以设置，但实际游戏中没有计时器。

**后续实现**：
- 每步走棋时启动步时倒计时，超时判负
- 总用时超过局时判负
- 计时器在服务端运行以保证权威性

## 8. 无法开始对局

**描述**：
> 1. 对战人数满了以后会一直显示等待对手加入，无法正常对局
> 2. 房主开房之后没有（悔棋/步时/局时/先后手）等设置，也没有准备和开始游戏选项
> 3. 后进入的玩家进房之后看不到（悔棋/步时/局时/先后手）等房间设置描述文本，也没有准备选项
